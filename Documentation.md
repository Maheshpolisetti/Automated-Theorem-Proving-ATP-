# Set Theory Automated Theorem Prover: Guide

The project includes a fully robust graphical interface designed to simplify the evaluation process and provide an intuitive visualization of the resulting proof paths. 

To launch the graphical interface, run:
```bash
python app_atp.py
```

## Layout Overview
- **Left Panel (Control & Input):** This panel is where the logic environment is established. It features dropdowns for pre-loaded mathematical examples, an interactive Symbol Palette for typing math characters smoothly, and execution controls.
- **Right Panel (Proof Graphing):** Once a proof executes, this panel generates an interactive `networkx` state-graph rendering the full resolution step tree leading down to the contradiction (`□`).

## Testing with Pre-Loaded Examples
If you want to immediately see the engine in action without typing your own formulas, the uppermost control on the Left Panel is the **Load Set Theory Example** dropdown. 
Selecting any option from this menu—such as **Subset Transitivity**, **Union Membership**, or the **Empty Set Check**—will automatically populate the Knowledge Base and Goal fields with the correct mathematical syntax. This is the fastest way to verify the engine's capabilities or understand how to properly structure your own proofs.

## Executing Custom Queries
The core function of the ATP is verifying conclusions against assumptions. The GUI splits this into two primary inputs:

1. **Knowledge Base (KB) Field:** The theorem prover is already hard-loaded with the 7 fundamental Set axioms. The KB text box is exclusively utilized to define **additional constraints** or the specific "If" premises of your query. 
   - *Example:* If you want to prove "If A is a subset of B, and B is a subset of C, then A is a subset of C", you would place the two established assumptions into this box on separate lines:
     `Subset(A, B)`
     `Subset(B, C)`
2. **Goal to Prove Field:** This acts as the final "Then" statement of your hypothesis. You only place the explicit conclusion you want mathematically tested here.
   - *Example:* Following the prompt above, you place the conclusion here:
     `Subset(A, C)`

Once configured, click **RUN PROOF**. The engine will logically negate your Goal, map it against the KB assumptions + Set Axioms, process the combinatorial tree, and render the mathematical contradiction.

## Analyzing the Result (Viewing the Proof)
Once the execution finishes, the GUI delivers the output in two distinct formats simultaneously:

### 1. The Step-by-Step Textual Log
Located directly below the **RUN PROOF** button in the **Left Panel** is the scrollable text log. 
- It updates sequentially to print out the exactly derived CNF clauses mathematically generated by the resolution engine.
- Every mathematically distinct logic step (like deriving an intermediate fact from two parent axioms) is printed here, culminating in the final derivation step.

### 2. The Visual Proof Graph
Located taking up the entire **Right Panel** is the interactive canvas.
- When the theorem reaches a contradiction, a hierarchical **directed graph** is automatically drawn here.
- The diagram structurally connects the generated "parent" clauses down into their final unified mathematical conclusion (resolvent). 
- **Green Node:** Represents the final empty clause (`□`) which successfully triggers the contradiction.
- **Blue Nodes:** Represent the initial starting logic (your foundational Set Theory Axioms, the KB constraints you entered, and your parsed Goal). These are the roots of the derivation.
- **Orange Nodes:** Represent mathematically derived intermediate formulas (Sub-clauses). Whenever the engine resolves two parent nodes together to find a new logical truth, that new generated truth is plotted as an Orange node.
